# Relay Hub 通用性审计

这份文件只回答两件事：

1. 哪些层是通用的
2. 当前 `main-Windows` 分支到底交付到了哪一步

## 1. AI 宿主协议层

协议层是通用的。一个对象只要满足下面 3 个条件，就可以接入：

1. 能执行本地命令
2. 能长期保留一小段最小规则
3. 能读写普通文件

因此，协议层并不绑定某个固定宿主。

但“协议层通用”不等于“所有宿主都已经现成接好”。  
真正的接入完成仍取决于该宿主是否已经把：

- 长期规则
- `main_session_ref` 规则
- pickup 启动链路
- 自动精确镜像机制

真正落下并验证。

## 2. OpenClaw 消息层

消息层的边界是固定的：

- 支持：`OpenClaw` 已支持的消息渠道
- 不支持：绕过 `OpenClaw` 直接读写原始渠道
- 不支持：没有 `OpenClaw` 的独立网关模式

也就是说，Relay Hub 的消息网关模型是：

- 通用到“任意 OpenClaw 渠道”
- 不通用到“任意非 OpenClaw 网关”

## 3. 平台层

协议和脚本本身大体是通用的，但**当前分支现成交付的安装与托管路径只面向 Windows**。

当前分支已经整理好的，是：

- Windows 共享安装层
- Windows 宿主 Web 后台托管
- Windows 安装文档与运行文档

当前分支没有对外承诺的，是：

- 其他操作系统的现成安装路径
- 所有 AI CLI 的现成 adapter
- 所有消息网关的零改动通用接入

## 4. 当前最准确的结论

`main-Windows` 当前可以如实宣称的是：

- Relay 协议层是通用的
- OpenClaw 渠道模型是通用的
- Windows 安装主路径已经单独整理出来

`main-Windows` 当前不能宣称的是：

- “拿给任意 AI 编程工具和任意 OpenClaw 实例就一定已经实机可用”
- “Windows + OpenClaw + 真实消息渠道已经端到端联调完成”
- “所有宿主都不需要再补自己的长期机制”

更准确地说：

- Windows 宿主侧支持可以单独落地
- OpenClaw 侧接入也可以单独落地
- 真实消息渠道端到端是否成立，仍要看后续真实 OpenClaw 实例和消息渠道联调结果
